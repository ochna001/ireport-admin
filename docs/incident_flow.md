# Incident Lifecycle & Data Flow

This document describes the end-to-end flow from the moment a resident submits a report in the field app to the final report generated by the admin/incident desk. It focuses on how the database is updated at each step and the actors involved.

## 1. Resident submits a report (Field App)

**Actor:** Resident or Reporter via `Ireport` Android app

**API:** `SupabaseIncidentsRepository.createIncident` (field app) inserts into `incidents`

**DB writes:**
- `incidents`
  - `status = 'pending'`
  - `agency_type` (PNP/BFP/PDRRMO) based on the app form
  - `reporter_id`, `reporter_name`, `reporter_age`, `reporter_phone`
  - `description`, `location_address`, `latitude`, `longitude`
  - `media_urls` (if the field app attaches evidence)
  - timestamps: `created_at`, `updated_at`

At this point there is **no station assignment** yet, and the incident remains pending.

## 2. Auto-assign to nearest station (Backend IPC `db:updateIncidentStatus`)

**Trigger:** When the incident transitions from `pending` to `assigned` or `in_progress` in the Admin App (`IncidentDetail.tsx`).

**Logic:**
- `db:updateIncidentStatus` (Electron main process @src/main/index.ts#300+) detects if `assigned_station_id` is null.
- Looks up the agency (`agencies.short_name`) and all stations for that agency
- Calculates Haversine distance from incident lat/lng to each station
- Sets `incidents.assigned_station_id` to closest station ID
- If admin explicitly chooses a station, that overrides the auto assignment.

**DB writes:**
- `incidents.assigned_station_id`
- `incident_status_history` entry with status change
- `first_response_at` set when moving off pending

## 3. Station notified and receives report

**Actor:** Station chief / desk officer in admin app

**Mechanism:**
- The assigned station sees only their incidents (scoped via session in `getIncidents`, `getStats`, etc.).
- Notifications are created when the incident is assigned to officers (later step), but station staff sees new incidents in their dashboard/list.

**DB read scopes:**
- `sessionScope.stationId` filters incidents (`getIncidents`, `getStats`, `exportIncidents`)

## 4. Station chief/desk officer reviews the report (Admin App)

**UI:** `IncidentDetail.tsx`

**Actions:**
- Review initial report (Reporter narrative)
- Review media attachments
- Review map location, timeline, and unit reports

**No DB writes** yet unless they change status or assignments.

## 5. Assign responders / field officers

**UI:** `IncidentDetail.tsx` -> status & assignment panel

**IPC:** `db:updateIncidentStatus`

**DB writes when officers/resources selected:**
- `incidents.assigned_officer_ids` and `assigned_officer_id` (primary)
- `incidents.assigned_resource_ids`
- `profiles.status` set to `busy` for newly assigned officers, back to `available` when unassigned or when incident resolves
- `agency_resources.status` set to `deployed` and back to `available`
- `incident_status_history` entry and optional `incident_updates` note
- `notifications` entries created for each officer assigned (`New Incident Assignment`)

## 6. Responders notified and receive the report

**Actors:** Field officers / responders (mobile app)

**Mechanism:**
- Admin app inserts rows in `notifications` for each officer
- Mobile app polls Supabase (or receives push notifications via `notifications` + `push_tokens` if enabled)
- Field officers see the assigned incident via `incidentsRepository` (mobile app)

**DB reads:**
- `incidents` filtered by assigned IDs in mobile app
- `notifications` table if push/alerts implemented

## 7. Responders conduct investigation and prepare final report draft

**Actors:** Responders in mobile app, or station admin collecting details

**Current Implementation:**
- Mobile app currently marks incident as `resolved` after submitting form (PNP/BFP/MDRRMO form). However, final report drafting still occurs in admin flow when closing the incident.
- If responders need to submit a draft, they can add `unit_reports` entries (Field app already writes unit reports via `unit_reports` table).

**DB writes:**
- `unit_reports`: contains `title`, `details`, `responder_id`, `agency`
- `incident_status_history` when mobile sets status to `resolved`

## 8. Submit final report & close incident (Admin)

**UI:** `IncidentDetail.tsx` -> "Close Incident" triggers final report modal

**Data captured:**
- `FinalReportData` (summary, actionsTaken, outcome, recommendations, plus agency-specific fields like case numbers, suspects, fire origin, estimated damage, casualties, assistance provided, etc.)
- Completed by current admin user (`completedBy = scope.userId`)

**IPC:** `finalReports:create`

**DB writes:**
- `final_reports`
  - `incident_id`
  - `report_details` JSONB storing all fields
  - `completed_by_user_id`
  - `completed_at`
- `incidents.status` updated to `closed`, `resolved_at` set
- `incident_status_history`: entry for `closed`
- Officers/resources freed (`profiles.status = available`, `agency_resources.status = available`)
- `notifications` can be sent to reporter if desired (not currently implemented)

## 9. Timeline Summary of Table Updates

| Step | Table(s) Updated | Key Columns |
| --- | --- | --- |
| Resident Report | `incidents`, `media` (if mobile uploads) | initial incident data |
| Auto Station Assignment | `incidents` | `assigned_station_id`, `first_response_at` |
| Officer Assignment | `incidents`, `profiles`, `agency_resources`, `notifications`, `incident_status_history`, `incident_updates` | assignment arrays, statuses, notes |
| Field Updates / Unit Reports | `unit_reports` | `details`, `responder_id`, `created_at` |
| Final Report & Closure | `final_reports`, `incidents`, `incident_status_history`, `profiles`, `agency_resources` | final report JSON, status closure, freeing resources |

## Notes

- **Media**: Resident-submitted media lives in `incidents.media_urls`. Admins can upload additional evidence via the new `media` table (Supabase Storage).
- **Notifications**: Currently triggered for officers when added to an incident. Push notifications to residents/responders require `push_tokens` integration.
- **Station Scope**: Chiefs automatically filter to their station and agency across dashboard, incidents, reports, and exports via `sessionScope`.
